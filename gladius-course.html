01

From Request to Working Code

What Gladius does and how your request becomes reality

What is Gladius?

Imagine you could describe a coding task in plain English — like "add login to the homepage" — and have AI agents automatically plan it, code it, and review it for you. That's Gladius.

Gladius is a multi-agent AI system that coordinates three specialized agents to handle software development tickets end-to-end:

📋
Planner

Analyzes your request and creates a step-by-step implementation plan

💻
Implementer

Writes the actual code changes based on the plan

👁️
Reviewer

Checks the code for quality, correctness, and security issues

The Journey of a Request

Let's trace what happens when you type: "Add input validation to the login form"

1
You submit your request

Type it in plain English or use a ticket ID from Jira. Gladius figures out what you mean.

2
Planner analyzes the task

Reads your request, understands the context, and breaks it into concrete steps: "Add regex validation, show error messages, add tests"

3
Implementer writes the code

Takes the plan and generates actual code changes — new functions, modified files, test cases

4
Reviewer checks the work

Looks for bugs, security issues, and quality problems. If it finds critical issues, sends feedback to Implementer for a revision

5
You get the results

A complete audit trail: the plan, the code changes as patches, the review verdict, and all the feedback

Under the Hood: The Entry Point

When you run gladius "add validation", here's the first code that runs:

CODE 
def run_pipeline(
    request: str,
    model: str = "sonnet",
    project_path: Path = None
):
    # Get ticket data
    ticket_data = mcp_client.get_ticket(request)

    # Run three agents in sequence
    plan = planner.execute(ticket_data)
    code = implementer.execute(plan)
    review = reviewer.execute(code)
PLAIN ENGLISH

Define a function that takes your request, which AI model to use, and where your code lives...

Convert your plain English request into a structured ticket that agents can understand...

Hand the ticket to Planner, who creates a step-by-step plan...

Give the plan to Implementer, who writes the actual code...

Send the code to Reviewer, who checks it for problems and decides whether to approve it or request changes.

💡
Key Insight

Each agent runs independently — they don't share memory or state. They only communicate through JSON messages, like coworkers passing notes. This makes the system reliable — if one agent fails, it doesn't crash the others.

Meet the Three Agents

The specialized AI workers that make Gladius work

Three Roles, One Goal

Think of Gladius as a tiny software team where each member has one job and does it really well. They don't step on each other's toes — they collaborate by passing information forward.

📋

Planner Agent

The Architect. Reads your request, figures out what needs to change, and creates a detailed plan with steps, files to modify, and risks to watch for.

💻

Implementer Agent

The Builder. Takes the plan and writes the actual code — new functions, modified files, tests. Outputs everything as a unified diff patch you can apply.

👁️

Reviewer Agent

The Quality Guard. Checks the implementation for bugs, security issues, and bad patterns. Approves good code or sends it back with specific feedback for revision.

💬
Why Three Agents?

Separating planning, implementation, and review mirrors how real engineering teams work. It's also how you'd instruct an AI assistant — "first make a plan, then code it, then check your work." Gladius automates that whole conversation.

How Agents Are Built

Every agent follows the same pattern. Here's the Planner agent's structure:

CODE 
class PlannerAgent:
    def execute(self, context):
        # Build the message for Claude
        message = self.build_user_message(context)

        # Invoke Claude CLI with JSON schema
        result = claude_client.invoke(
            message,
            json_schema=PlannerOutput.schema()
        )

        # Validate and return
        return PlannerOutput(**result)
PLAIN ENGLISH

Define the agent class with an execute method that takes context data...

Construct a detailed prompt with the ticket info and any additional context...

Call Claude (the AI model) via the CLI, giving it a JSON schema that defines exactly what fields the response should have...

Take Claude's JSON response and validate it against our schema to make sure it has all the required fields (summary, plan, files to change, etc.).

💡
Key Insight

The JSON schema is critical. It tells Claude exactly what format to return, so Gladius can reliably parse the response. Without it, Claude might write an essay instead of structured data.

Splitting work across three agents has real benefits:

✓
Better Quality

✓
Easier to Debug

If something goes wrong, you know which agent to fix. Bad plan? Fix Planner. Bad code? Fix Implementer.

✓
Extensible Design

Want to add a "Tester" agent or "Documentation" agent? Just plug them into the pipeline.

✓
Mirrors Real Teams

Architects, developers, and code reviewers — this structure reflects how human teams already work

03

How Agents Talk

Stateless communication through structured messages

Like a Postal System

Imagine a postal service where each worker operates independently — the mail sorter doesn't peek into the delivery driver's truck, and the delivery driver doesn't know what's sitting on the sorter's desk. They only communicate by passing sealed packages with clear labels.

That's how Gladius agents work. Each agent is stateless — they don't remember past conversations or share a workspace. They only know what's in the message they receive.

📦
Each message is complete

Planner gets the ticket. Implementer gets the ticket AND the plan. Reviewer gets ticket + plan + code.

🔄
No shared memory

Agents don't access each other's variables or state — only the JSON messages passed between them

✨
Fresh start every time

Each agent invocation is a new subprocess. No lingering bugs from previous runs.

What's in a Message?

Every message between agents is JSON — structured data with labeled fields. Here's what Implementer receives:

JSON MESSAGE 
{
  "ticket": {
    "key": "PROJ-123",
    "title": "Add authentication",
    "description": "..."
  },
  "plan": {
    "summary": "Add JWT auth...",
    "plan": [
      "Create auth.py",
      "Add middleware",
      "Write tests"
    ],
    "files_to_change": [...]
  },
  "project_path": "/my/project"
}
WHAT IT MEANS

The original user request, converted into structured ticket data with a unique ID...

The plan that Planner created — summary, step-by-step instructions, and which files need changes...

Where the codebase lives, so Implementer knows which project to work on.

This complete message is everything Implementer needs. It doesn't need to ask Planner questions or look at Planner's internal state — it's all here.

Why Build It Stateless?

Making agents stateless might seem like extra work — why not let them share memory? Here's why it's worth it:

🛡️

Isolation

If Reviewer crashes, it doesn't corrupt Planner or Implementer's state. Each agent fails independently.

🔁

Retryable

If an agent fails, you can retry it with the exact same input. No "it worked last time" mysteries.

📝

Auditable

Every message is saved as JSON. You can replay the entire pipeline by reading the saved messages.

⚖️

Parallelizable

No shared state means agents could run in parallel (if the workflow allowed it) without stepping on each other.

💡
Key Insight

Stateless design is a trade-off. You send more data with each message (because each agent gets a complete context), but you gain reliability and simplicity. It's the same principle behind REST APIs and functional programming — avoid hidden state, make data flow explicit.

Building the Context Message

Here's how the Orchestrator prepares the message for Implementer:

CODE 
def _run_implementer(
    self, ticket_data, plan_output,
    review_output, iteration, model, project_path
):
    context = {
        "ticket": ticket_data,
        "plan": plan_output,
        "model": model,
        "iteration": iteration,
        "project_path": str(project_path)
    }

    # Add review feedback if revision
    if review_output:
        context["review_feedback"] = {
            "issues": review_output.issues,
            "suggested_changes": review_output.suggested_changes
        }

    return implementer.execute(context)
PLAIN ENGLISH

Create a dictionary (like a labeled box) with everything Implementer needs: the ticket, the plan, which AI model to use, which attempt this is, and where the code lives...

If this is a revision (Reviewer sent it back), add the review feedback — what issues were found and what changes were suggested...

Call Implementer's execute method with this complete context. Implementer starts fresh, knowing only what's in this message.

Notice: the Orchestrator doesn't tell Implementer "remember what you did last time." It explicitly includes the review feedback in the message. Everything is in the data, nothing is in shared memory.

04

The Claude CLI Worker

How Gladius talks to the AI model

The Assembly Line

Think of the Claude CLI as a specialized machine on a factory floor. You feed it raw materials (a prompt and a JSON schema), press the button (run the command), wait while it works, and receive a finished product (structured JSON output).

Gladius doesn't use the Claude API directly. Instead, it invokes the Claude CLI as a subprocess — a separate program that runs, does its job, and returns a result.

1
Build the command

Construct a shell command with flags: claude --print --model sonnet --json-schema {...}

2
Run the subprocess

Execute the command, showing a spinner while Claude thinks (can take 30-120 seconds)

3
Capture the output

Read the text that Claude prints to stdout

4
Parse the JSON

Extract the structured data from Claude's response, handling wrapper formats and markdown

5
Validate the schema

Check that the JSON has all required fields using Pydantic

Invoking the CLI

Here's the core function that calls Claude:

CODE 
def invoke(self, user_message, system_prompt,
         json_schema, model="sonnet"):
    cmd = [
        self.claude_path,
        '--print',
        '--no-session-persistence',
        '--model', model,
        '--output-format', 'json',
        '--json-schema', json.dumps(json_schema),
        user_message
    ]

    result = subprocess.run(
        cmd,
        stdout=subprocess.PIPE,
        text=True,
        timeout=self.timeout
    )

    return self._parse_json_output(result.stdout)
PLAIN ENGLISH

Build a command as a list of arguments: the path to Claude, flags to control its behavior, the AI model to use, and importantly the JSON schema that defines what structure the response should have...

Run the command as a subprocess, capturing its text output (stdout), with a timeout to prevent hanging forever...

Take the raw text output and parse it into structured JSON data we can work with.

🔍
Why the CLI Instead of the API?

The Claude CLI handles authentication, retries, and rate limiting automatically. It also integrates with MCP (Model Context Protocol) for accessing external data. Using the CLI means less code to maintain and better integration with the Claude ecosystem.

Parsing JSON (The Hard Part)

You'd think "Claude outputs JSON" means it's easy to parse. Not quite. Claude sometimes wraps the JSON in different formats:

1️⃣

Direct JSON

Best case: {"summary": "..."} — clean, parseable JSON

2️⃣

Markdown Wrapper

```json\n{"summary": "..."}\n``` — need to extract from code block

3️⃣

CLI Wrapper

{"type": "result", "structured_output": {"summary": "..."}} — nested structure

4️⃣

Mixed Content

Text before/after JSON — need to extract just the JSON part

Gladius handles all these formats with a multi-strategy parser that tries each approach until one works.

Multi-Strategy Parsing

Here's the parsing logic that handles all those formats:

CODE 
def _parse_json_output(self, output):
    # Try direct parsing first
    try:
        parsed = json.loads(output)
        if "structured_output" in parsed:
            return parsed["structured_output"]
        return parsed
    except:
        pass

    # Try extracting from markdown
    match = re.search(
        r'```(?:json)?\s*(\{.*?\})\s*```',
        output,
        re.DOTALL
    )
    if match:
        return json.loads(match.group(1))

    # Find any JSON object in the text
    matches = re.findall(
        r'\{[^{}]*(?:\{[^{}]*\}[^{}]*)*\}',
        output
    )
    for m in matches:
        try:
            return json.loads(m)
        except:
            continue

    raise JSONDecodeError("No valid JSON found")
PLAIN ENGLISH

Strategy 1: Try parsing the entire output as JSON. If it's wrapped in a CLI structure, extract the "structured_output" field. This handles the clean cases...

Strategy 2: Look for JSON inside markdown code blocks (the triple backticks). Use a regex pattern to extract it...

Strategy 3: Search the entire text for anything that looks like a JSON object (starts with { and ends with }). Try parsing each match until one works...

If all strategies fail, raise an error — we couldn't find valid JSON anywhere.

💡
Key Insight

This multi-strategy parsing is defensive programming. Instead of assuming Claude will always return perfect JSON, we handle the reality that LLMs sometimes wrap their output in unexpected ways. The cost is a few extra parsing attempts; the benefit is reliability in production.

The Review Loop

How Reviewer improves code through feedback iterations

Like Editing a Draft

Imagine writing an essay. You write a first draft, then your editor reads it and marks it up: "This argument needs support," "This section is unclear," "Great insight here, expand it." You revise based on the feedback, and the editor reads it again. If it's good, they approve it. If not, another round.

That's exactly how Gladius's review loop works. Implementer writes code, Reviewer evaluates it, and if there are issues, Implementer gets specific feedback for a revision.

🔍
Reviewer checks for issues

Security bugs, incorrect logic, missing error handling — categorized as critical, major, or minor

✅
Two possible verdicts

APPROVE: Code is good, ship it. REQUEST_CHANGES: Fix these issues first.

🔄
Implementer revises

Gets the feedback, understands what needs fixing, rewrites the code addressing the issues

🛑
Maximum iterations

The loop runs up to 2-3 times by default, then stops even if not perfect (prevents infinite loops)

The Iteration Pattern in Code

Here's how the Orchestrator manages the review loop:

CODE 
approved = False
iteration = 1
review_output = None

while iteration <= max_iterations and not approved:
    # Implementer creates/revises code
    impl = implementer.execute(
        ticket, plan, review_output
    )

    # Reviewer evaluates the code
    review = reviewer.execute(
        ticket, plan, impl
    )

    # Check the verdict
    if review.verdict == "APPROVE":
        approved = True
    else:
        review_output = review  # Pass to next iteration
        iteration += 1

return approved, impl, review
PLAIN ENGLISH

Start with "not approved" and iteration 1, with no review feedback yet...

Keep looping as long as we haven't hit the max attempts AND the code isn't approved yet...

Implementer writes/revises code. On iteration 1, this is the first draft. On later iterations, Implementer gets the previous review feedback...

Reviewer evaluates the implementation, checking for bugs, security issues, and quality problems...

If Reviewer approves, we're done! Break out of the loop. If not, save the review feedback and increment the counter for the next round...

Return whether it was approved, the final code, and the final review.

💡
Key Insight

This is a bounded loop with an exit condition. Without max_iterations, you could get stuck forever if Implementer and Reviewer disagree. The limit ensures the pipeline always completes, even if the code isn't perfect.

What Reviewer Looks For

Reviewer categorizes issues by severity to help Orchestrator decide whether to iterate:

🚨

Critical

Blocking. Security vulnerabilities, data loss risks, broken functionality. Must be fixed.

⚠️

Major

Important. Missing error handling, poor architecture, significant bugs. Should be fixed.

ℹ️

Minor

Nice-to-have. Code style issues, small optimizations, documentation gaps. Can ship without fixing.

Reviewer typically requests changes only if there are critical or major issues. Minor issues get noted but don't block approval.

What Feedback Looks Like

When Reviewer requests changes, it sends structured feedback that Implementer can act on:

JSON FEEDBACK 
{
  "verdict": "REQUEST_CHANGES",
  "summary": "One critical security issue",
  "issues": [
    {
      "severity": "critical",
      "category": "security",
      "description": "JWT secret hardcoded",
      "location": "auth.py:15",
      "recommendation": "Read from env var"
    }
  ],
  "suggested_changes": [
    "Use os.environ['JWT_SECRET']",
    "Add .env file with secret",
    "Document env vars in README"
  ]
}
WHAT IT MEANS

The verdict is "needs work" — Implementer must revise...

A quick summary of what's wrong...

Each issue includes: how serious it is, what category (security, correctness, etc.), a description, where in the code it appears, and what to do about it...

Concrete suggestions for fixing the issues — actionable steps Implementer can follow.

On the next iteration, Implementer receives this feedback as part of its context and addresses each issue in the revision.

The Orchestrator

The conductor that coordinates the entire pipeline

The Air Traffic Controller

Imagine an air traffic controller at a busy airport. They don't fly the planes — pilots do that. But the controller coordinates everything: which plane lands first, where it taxis, when it takes off. They track each plane's status, handle delays, and ensure nothing crashes.

That's the Orchestrator in Gladius. It doesn't write code or create plans — the agents do that. But it manages the entire workflow:

1️⃣
Fetches the ticket

Gets your request from MCP or converts your natural language into a ticket

2️⃣
Runs agents in sequence

Calls Planner → Implementer → Reviewer in order, passing outputs as inputs

3️⃣
Manages the review loop

Handles iterations between Implementer and Reviewer until approval or max attempts

4️⃣
Saves artifacts

Writes all outputs (ticket, plan, code, review) to disk with timestamps for auditing

The Pipeline Flow

Here's the core orchestration logic:

CODE 
def run_pipeline(self, request, model, project_path):
    # Phase 1: Fetch ticket
    ticket_data = mcp_client.get_ticket(request)
    run_dir = artifact_manager.create_run(ticket_data["key"])

    # Phase 2: Planning
    plan = planner.execute({
        "ticket": ticket_data,
        "project_path": project_path
    })
    artifact_manager.save_planner_output(plan)

    # Phase 3: Implementation + Review loop
    approved = False
    iteration = 1
    review = None

    while iteration <= max_iterations and not approved:
        impl = implementer.execute({
            "ticket": ticket_data,
            "plan": plan,
            "review_feedback": review
        })
        artifact_manager.save_implementer_output(impl, iteration)

        review = reviewer.execute({
            "ticket": ticket_data,
            "plan": plan,
            "implementation": impl
        })
        artifact_manager.save_reviewer_output(review, iteration)

        approved = (review.verdict == "APPROVE")
        iteration += 1

    # Phase 4: Create summary
    summary = self._create_summary(ticket_data, plan, impl, review)
    artifact_manager.save_run_summary(summary)

    return summary
PLAIN ENGLISH

Get the ticket data and create a timestamped run directory where all outputs will be saved...

Run Planner with the ticket and project path. Save its output (the plan) to disk...

Enter the implementation/review loop. Start at iteration 1 with no review feedback...

Keep looping: Implementer writes code (using review feedback if available), save the code, Reviewer evaluates it, save the review. If approved, stop. Otherwise, increment the counter and loop again...

When the loop exits, create a summary with metrics (duration, files changed, issues found, approval status) and save it...

Return the summary to the user.

💡
Key Insight

The Orchestrator is the only component that understands the full workflow. Each agent only knows its own job. This separation means you can change the pipeline (add agents, reorder steps, change loop logic) without touching agent code.

Saving Artifacts

Every run creates a complete audit trail on disk:

FILE STRUCTURE 
runs/PROJ-123_20260330_143052/
├── ticket.json
├── planner/
│   └── plan.json
├── implementer/
│   ├── implementation_v1.json
│   └── implementation_v2.json
├── reviewer/
│   ├── review_v1.json
│   └── review_v2.json
├── patches/

│   └── changes_v2.patch

WHY EACH FILE EXISTS

The original ticket data — what the user requested...

Planner's output — the implementation plan...

Each iteration's code changes. v1 is the first try, v2 is the revision after review feedback...

Each review's verdict and issues. Reviewer checks v1, then checks v2 if changes were requested...

Git-compatible patch files you can apply to your codebase with git apply...

Summary of the entire run — duration, success/failure, metrics.

This structure means you can replay the entire pipeline by reading these files. It's also invaluable for debugging — if something went wrong, you know exactly what each agent saw and produced.

The Big Picture

Connecting Gladius to real-world patterns and your future projects

You Now Understand

You've seen how Gladius turns a natural language request into working code through a coordinated multi-agent system. Let's connect these ideas to the broader software world:

🏗️

Pipeline Architecture

Sequential stages where each step transforms data and passes it forward — used in CI/CD, data processing, and build systems

🔄

Stateless Design

Components that don't remember previous runs — makes systems more reliable and easier to scale (REST APIs, serverless functions)

💬

Message Passing

Explicit data flow instead of shared memory — the foundation of microservices and distributed systems

🎯

Separation of Concerns

Each component has one clear job — easier to understand, test, and modify (the Unix philosophy: "do one thing well")

🌍
These Patterns Are Everywhere

Gladius isn't unique — it applies established software patterns to AI agents. Understanding these patterns helps you recognize them in other systems and use them in your own projects.

When to Build Multi-Agent Systems

Not every problem needs multiple agents. Here's when it makes sense:

✓
The task has distinct phases

Planning → Execution → Review. If you'd naturally say "first do X, then do Y, then check Z," that's a pipeline.

✓
Specialists improve quality

A "Reviewer" agent catches mistakes an "Implementer" misses. Specialization adds value.

✓
You need auditability

Saving each agent's output creates a paper trail. Great for debugging and compliance.

✗
When NOT to use it

Simple tasks where one AI call would work. Over-engineering a solution adds complexity without benefit.

Principles for Steering AI

Whether you're using Gladius or building your own AI tools, these principles will help you get better results:

1️⃣
Be Specific About Structure

Don't just ask for "a plan." Ask for "a JSON object with a summary, a list of steps, and files to change." Structured outputs make AI responses usable, not just readable.

2️⃣
Design for Failure

AI isn't perfect. Build in retries (Gladius tries 3 times), validation (check the JSON schema), and bounded loops (max iterations). Expect things to go wrong and handle it gracefully.

3️⃣
Make Data Flow Explicit

Avoid "the AI remembers context." Pass complete messages. If Implementer needs the plan, send it explicitly. This makes debugging trivial — you can see exactly what each agent received.

4️⃣
Start Simple, Add Agents As Needed

Don't build a 5-agent system on day one. Start with one AI call. If quality suffers, add a Reviewer. If planning is weak, add a Planner. Let real problems drive the architecture.
